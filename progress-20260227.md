# Progress Log — 2026-02-27

## Implement Missing neat-python Features (Phases 1-3)

### Summary
Implemented three phases of missing neat-python features in NEAT.jl: RecurrentNetwork evaluator, utility functions (softmax/tmean), and Checkpointer reporter.

### Phase 1: RecurrentNetwork Evaluator
- **Files created:** `src/recurrent.jl`, `test/test_recurrent.jl`
- **Files modified:** `src/NEAT.jl` (include + exports)
- **What:** `RecurrentNetwork` struct that handles cyclic genomes (`feed_forward=false`) by maintaining internal state across activations. Uses double-buffering: reads from `prev_values`, writes to `values`, then swaps. Nodes evaluated in sorted ID order for determinism.
- **Why:** The library already supported recurrent genomes in the genetic operators (self-connections, cycle-allowing mutations), but had no network evaluator that could handle them. Anyone using `feed_forward=false` would get broken evaluation.
- **Design decision:** Matches neat-python's `neat/nn/recurrent.py` approach exactly — flat evaluation order with previous-timestep reads. This means multi-layer acyclic graphs take 2+ activations to propagate (unlike FeedForwardNetwork which uses topological ordering). This is correct recurrent behavior.
- **Tests:** 32 tests including hand-computed multi-timestep values with self-connections and cycles, FF equivalence for single-layer graphs, and multi-layer convergence verification.

### Phase 2: Utility Functions (softmax, tmean)
- **Files modified:** `src/utils.jl`, `src/validation.jl`, `src/NEAT.jl`, `test/runtests.jl`
- **What:** Added `softmax()` (numerically stable, subtracts max before exp), `tmean()` (trimmed mean with configurable trim fraction), and registered `:tmean` as a valid species fitness stat function.
- **Why:** neat-python's `math_util.py` provides both. `softmax` is broadly useful for probability distributions from network outputs. `tmean` provides robust fitness statistics less sensitive to outliers.
- **Tests:** Sum-to-1, uniform inputs, numerical stability with large values, known trimmed mean results, empty collection error.

### Phase 3: Checkpointer Reporter
- **Files created:** `src/checkpointer.jl`, `test/test_checkpointer.jl`
- **Files modified:** `src/NEAT.jl` (include + exports + Serialization import), `Project.toml` (Serialization dep)
- **What:** `Checkpointer <: Reporter` that saves complete evolution state at configurable intervals (generation count or wall-clock time). `CheckpointData` struct captures everything needed to reconstruct a `Population`. `save_checkpoint()`/`restore_checkpoint()` for manual use. `Population(data::CheckpointData)` constructor for restoration with automatic counter adjustment.
- **Why:** Long evolution runs (100-500 generations for pendulum examples) can't be interrupted/resumed without losing all progress.
- **Design decision:** Uses Julia's `Serialization` stdlib for fast, complete state capture. This is version-specific, which is acceptable for same-session resume. Cross-version portability is already available via the JSON export functions.
- **Tests:** 13 tests covering round-trip, generation interval correctness, continued evolution, counter preservation (no ID conflicts), species preservation, and manual save.

## CTRNN and Izhikevich Spiking Network Evaluators

### Summary
Implemented CTRNN and Izhikevich spiking neural network evaluators, with the necessary NodeGene/GenomeConfig extensions and JSON round-trip support.

### Phase 1: NodeGene and GenomeConfig Extension
- **Files modified:** `src/genes.jl`, `src/config.jl`, `src/validation.jl`, `src/genome.jl`
- **Files created:** `test/test_ctrnn_iznn_genes.jl`
- **What:** Added 5 fields to `NodeGene`: `time_constant` (CTRNN), `iz_a`, `iz_b`, `iz_c`, `iz_d` (Izhikevich). All default to `NaN`, making them invisible to existing code paths. `GenomeConfig` conditionally creates `FloatAttribute` instances when `time_constant_init_mean` or `iz_a_init_mean` appear in config. Updated `init_attributes!`, `mutate!`, `copy`, `crossover`, `distance`, and `_node_distance_simple` with conditional handling.
- **Why:** Both CTRNN and IZNN need per-node evolvable parameters. Doing this in one pass avoids touching NodeGene twice.
- **Design decision:** `NaN` sentinel value instead of `Union{Nothing, Float64}` because Float64 fields avoid boxing and maintain struct layout efficiency. `isnan()` checks are fast and well-supported.
- **Tests:** 1043 tests including init/mutate with configured and unconfigured attributes, copy preservation, crossover inheritance ratios, and distance with/without configured fields.

### Phase 2: CTRNN Evaluator
- **Files created:** `src/ctrnn.jl`, `test/test_ctrnn.jl`
- **What:** `CTRNNNetwork` with double-buffered state, forward Euler integration: `y += (dt/τ) * (-y + z)` where `z = activation(bias + response * aggregation(weighted_inputs))`. Per-node evolvable time constants (extending neat-python's single global tau). `advance!()`, `reset!()`, `set_node_value!()`.
- **Why:** CTRNNs model continuous temporal dynamics used in control tasks. Per-node tau allows evolving heterogeneous temporal responses.
- **Tests:** 34 tests including hand-computed single Euler step, steady-state convergence, per-node tau effect verification, reset, variable dt handling, determinism, sigmoid activation, and multi-node recurrent topology.

### Phase 3: Izhikevich Spiking Network Evaluator
- **Files created:** `src/iznn.jl`, `test/test_iznn.jl`
- **What:** `IZNeuron` with half-step voltage updates, overflow protection, spike-based communication. `IZNNNetwork` with `set_inputs!()`, `advance!()`, `reset!()`. 7 preset parameter sets (regular spiking, fast spiking, bursting, etc.).
- **Why:** Enables experiments with temporal coding and biologically realistic spiking dynamics. Different (a,b,c,d) parameter sets produce distinct behaviors.
- **Design decision:** Half-step voltage updates match neat-python for numerical stability. `isfinite()` guard handles Julia's Inf/NaN behavior (vs Python's raise). Bare `a,b,c,d` field names in `IZNeuron` (local scope), `iz_` prefix in genome/config layer.
- **Tests:** 34 tests including initial state, spike/no-spike verification, fast vs regular spiking rate comparison, overflow handling, genome construction, recurrent spike propagation, hand-computed spike timing, and preset constant access.

### Phase 4: JSON Export/Import Extensions
- **Files modified:** `src/export.jl`
- **What:** Conditional serialization of `time_constant` and `izhikevich` (nested dict with `iz_a/b/c/d`) in node export. Import reads them back when present. Both single-genome and population export updated.
- **Design decision:** Only emitted when not NaN, so standard genome JSON is unchanged.

- XOR example runs correctly

## Fix Pre-existing JSON Export Test Failures

- **File modified:** `test/runtests.jl`
- **Root cause:** Two test expectation mismatches with the v1.0 export format:
  1. Test expected `imported_conn.innovation == 1`, but the v1.0 format doesn't serialize innovation numbers. Import defaults to 0.
  2. Test accessed `data["genomes"][1]["fitness"]` at top level, but v1.0 format stores fitness inside `data["genomes"][1]["metadata"]["fitness"]`.
- **Fix:** Updated test expectations to match the actual format. All 6,003+ tests now pass with zero failures.
