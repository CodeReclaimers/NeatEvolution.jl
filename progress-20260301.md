**New tests added:**

10. **AbstractNetwork interface** — `input_nodes`, `output_nodes`, `num_inputs`, `num_outputs` return correct values
11. **Analytical solution comparison** — Compares Euler integration against analytical solution at multiple time points
12. **Time accumulation** — Three `advance!` calls with different durations, verifies `net.time_seconds` equals the sum
13. **Zero-link output node** — Output node with no connections converges to its bias value

**Key finding — double-buffered integration dynamics:**
During the analytical solution comparison test, discovered that the CTRNN's double-buffered Euler integration effectively doubles the time constant for nodes with only external input (no self-connection). Each buffer is updated every other step, so the active buffer makes N/2 Euler updates in N real steps. The effective analytical solution is `y(t) = z * (1 - exp(-t / (2*tau)))` rather than `y(t) = z * (1 - exp(-t / tau))`. This is inherent to the double-buffered scheme (matching neat-python's implementation) and does not affect steady-state convergence, only transient dynamics. The test was written to compare against the correct (double-buffered) analytical formula with tight tolerances (atol=1e-3), validating Euler accuracy.

**Decisions:**
- Used the adjusted analytical formula `y(t) = z*(1-exp(-t/(2*tau)))` rather than trying to work around the double-buffer effect. This accurately tests what the code actually does.
- Kept dt=0.0001 for the analytical test — small enough that Euler discretization error is negligible compared to the 1e-3 tolerance.
- All tests follow existing patterns: use `make_ctrnn_test_config()` and `make_ctrnn_node()` helpers, include `println` diagnostic output.

**Verification:** All 420 tests pass (64 CTRNN assertions, full suite green).

## Added Lorenz attractor CTRNN prediction example

**Commit:** `1966e5e` — Add Lorenz attractor CTRNN prediction example

**Files created:**
- `examples/lorenz_ctrnn/config.toml` — NEAT/CTRNN config (3 inputs, 3 outputs, feed_forward=false)
- `examples/lorenz_ctrnn/lorenz_ctrnn.jl` — Main script with RK4 integrator, normalization, evolution, optional CairoMakie visualization
- `examples/lorenz_ctrnn/README.md` — Documentation with expected performance and experimental results

**What it does:** Evolves a CTRNN to predict next-step Lorenz attractor state from current state. Hand-written RK4 integrates the Lorenz ODE (no OrdinaryDiffEq dependency). Trajectory is subsampled 10x (effective dt=0.1s) to produce 800 train / 200 test data points. Fitness is negative MSE in normalized [-1,1] space.

**Key design decisions and experimental findings:**

1. **10x subsampling (dt=0.01 → effective dt=0.1s):** Without subsampling, per-step state change is ~0.02 in normalized space, so a "copy the input" strategy achieves MSE ~0.1 trivially. With 10x subsampling, per-step change is 0.23-0.36, forcing the network to learn actual dynamics. Both approaches produced the same quality results, but subsampling runs 3x faster (800 vs 8000 training steps). Chose subsampling as default.

2. **Per-variable performance (standard mode, 300 gen):**
   - x: correlation 0.90-0.96 (well tracked, approximately linear dynamics)
   - y: correlation 0.39-0.82 (partially learned, high variance across runs)
   - z: correlation ~0.00 (not learned — outputs attractor mean ~25)

3. **Why z fails:** The z equation dz/dt = xy - βz requires a bilinear term (product xy) that small networks with additive aggregation cannot easily represent. Evolved networks (3-10 nodes) never discover a multiplication-like circuit.

4. **`--products` flag experiment:** Augments inputs with pairwise products (xy, xz, yz), giving 6 inputs instead of 3. Results over 3 trials each:
   - y improves consistently: mean corr 0.66 → 0.85 (the y equation dy/dt = x(ρ-z) - y contains xz)
   - z remains ~0.00 in both modes despite xy being directly available as input
   - Overall MSE improves due to y gains

5. **Bumping node_add_prob (0.2 → 0.5 → 0.8):** Produced larger networks (up to 21 nodes, 61 connections) but worse overall MSE. Bigger networks are harder to optimize with the same population/generations. Kept default at 0.2.

6. **Config choices:** bias/weight range [-5,5] (narrower than default because activation is tanh(2.5z), saturates quickly), time_constant range [0.01, 5.0] with mean 1.0 (Lorenz orbital period ~1.5s), max_stagnation=30 (harder task needs patience).

**Alternatives rejected:**
- Using OrdinaryDiffEq: adds heavy dependency, hand-written RK4 is ~15 lines
- Per-example Project.toml: existing examples don't use one
- Starting with num_hidden > 0: tested, doesn't help; NEAT should discover topology
- Larger populations or more generations for z: diminishing returns, honest result is more informative

## Added --z-only flag; identified fitness dilution as primary z bottleneck

**Commit:** `014b2a7` — Add --z-only flag and document experimental findings

**Files modified:**
- `examples/lorenz_ctrnn/lorenz_ctrnn.jl` — Added `--z-only` flag, generalized eval/test/visualization to handle variable output counts via `target_rows` parameter
- `examples/lorenz_ctrnn/README.md` — Documented all four flag combinations with experimental results

**Key finding — z-prediction failure has two independent causes:**

The `--z-only` flag switches from 3 outputs (x,y,z) to 1 output (z only), with all fitness focused on z. This revealed that the previous diagnosis ("representation problem") was only half right:

1. **Fitness dilution (larger effect):** In 3-output mode, easy x/y gains dominate the fitness landscape. Evolution has no incentive to improve z because the same mutation effort yields larger MSE reductions on x and y. With z-only + standard 3 inputs: z correlation jumps from ~0.00 to 0.54-0.85.

2. **Representation difficulty (smaller but real effect):** The bilinear xy term is hard for additive-aggregation networks. With z-only + product inputs: z correlation reaches 0.94-0.96 with tiny networks (often 1 hidden node, 4 connections).

**Implementation notes:**
- `make_eval_genomes` and `evaluate_on_test` now take a `target_rows::Vector{Int}` parameter specifying which data rows are prediction targets (e.g., `[1,2,3]` for x,y,z or `[3]` for z-only)
- `load_lorenz_config` accepts both `num_inputs` and `num_outputs` kwargs, round-tripping through a temp TOML to work around immutable GenomeConfig
- Visualization adapts: 3D phase portrait only shown in 3-output mode; time series subplot count matches output count
- All four flag combinations (`--products` × `--z-only`) tested and working

## Added --mode flag with product-agg option; NEAT discovers multiplication

**Commit:** `84f59db` — Add --mode flag with base/products/product-agg options to Lorenz example

**Files modified:**
- `examples/lorenz_ctrnn/lorenz_ctrnn.jl` — Replaced `--products` flag with `--mode base|products|product-agg`, generalized `load_lorenz_config` to accept arbitrary genome overrides dict
- `examples/lorenz_ctrnn/README.md` — Rewrote with full experimental results tables including wall-clock times across all 6 mode×output combinations

**What changed:**

The `--products` flag was replaced by `--mode base|products|product-agg`. The new `product-agg` mode keeps the standard 3 inputs (x,y,z) but adds `:product` to the aggregation function options (alongside `:sum`) with 10% mutation rate. This lets evolution discover multiplicative nodes internally rather than requiring pre-computed product inputs.

**Implementation:**
- Added `parse_mode(args)` function to extract `--mode` value from ARGS
- Generalized `load_lorenz_config` from specific `num_inputs`/`num_outputs` kwargs to a `genome_overrides::Dict{String,Any}` parameter — any `[DefaultGenome]` key can now be overridden at runtime
- For product-agg mode, overrides are `aggregation_options=["sum", "product"]` and `aggregation_mutate_rate=0.1`
- `:product` aggregation already exists in NeatEvolution's aggregation registry (confirmed in `src/aggregations.jl:38`)

**Experimental results (3 trials each, 300 generations, population 150):**

3-output mode:

| Mode | x corr | y corr | z corr | MSE | Nodes | Time |
|------|--------|--------|--------|-----|-------|------|
| base | 0.84-0.96 | -0.09-0.82 | 0.00-0.17 | 0.10-0.12 | 4-10 | 12-18s |
| products | 0.93-0.97 | -0.17-0.66 | 0.00-0.12 | 0.11-0.12 | 6-17 | 13-20s |
| product-agg | 0.92-0.96 | 0.09-0.65 | 0.00-0.15 | 0.10-0.12 | 4-9 | 14-15s |

z-only mode:

| Mode | z corr | z MSE | Nodes | Time |
|------|--------|-------|-------|------|
| base | 0.83-0.86 | 0.055-0.069 | 2-3 | 9-10s |
| products | 0.94 | 0.023 | 1-3 | 10-12s |
| product-agg | 0.94 | 0.023-0.024 | 1-6 | 9-10s |

**Key finding:** The `product-agg` mode matches `products` mode exactly in z-only mode (both reach z corr 0.94). This means NEAT can discover the right computational primitive (multiplication) when it's available in the search space, achieving the same performance as hand-engineered product inputs. Networks are tiny in both cases (1-6 nodes).

**Two independent causes of z-prediction failure (refined from earlier analysis):**
1. **Fitness dilution (primary):** Switching from 3-output to z-only raises z corr from ~0.00 to 0.83-0.86 even with standard inputs.
2. **Representation difficulty (secondary):** Either pre-computed products or discoverable product aggregation raises z corr from 0.83-0.86 to 0.94 in z-only mode.

**Wall-clock times included** for future neat-python comparison. Times range from 9s (z-only base) to 20s (3-output products) on a single core.
