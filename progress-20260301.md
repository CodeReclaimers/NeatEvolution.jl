**Key finding — double-buffered integration dynamics:**
During the analytical solution comparison test, discovered that the CTRNN's double-buffered Euler integration effectively doubles the time constant for nodes with only external input (no self-connection). Each buffer is updated every other step, so the active buffer makes N/2 Euler updates in N real steps. The effective analytical solution is `y(t) = z * (1 - exp(-t / (2*tau)))` rather than `y(t) = z * (1 - exp(-t / tau))`. This is inherent to the double-buffered scheme (matching neat-python's implementation) and does not affect steady-state convergence, only transient dynamics. The test was written to compare against the correct (double-buffered) analytical formula with tight tolerances (atol=1e-3), validating Euler accuracy.

**Decisions:**
- Used the adjusted analytical formula `y(t) = z*(1-exp(-t/(2*tau)))` rather than trying to work around the double-buffer effect. This accurately tests what the code actually does.
- Kept dt=0.0001 for the analytical test — small enough that Euler discretization error is negligible compared to the 1e-3 tolerance.
- All tests follow existing patterns: use `make_ctrnn_test_config()` and `make_ctrnn_node()` helpers, include `println` diagnostic output.
