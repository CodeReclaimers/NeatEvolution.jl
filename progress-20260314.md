### Bug fixes

1. **CTRNN forward Euler reads wrong buffer** (`src/ctrnn.jl:164-165`)
   - Root cause: After the double-buffer swap (`active = 3 - active`), the old code did `ovalues[node_id] += (dt/tau) * (-ovalues[node_id] + z)`. Since `ovalues` points to the buffer that was written *two* steps ago (not the previous step), the integration used stale state. Invisible on the first step (both buffers initialized to zero) but produced incorrect dynamics afterward, effectively doubling the time constant.
   - Fix: Changed to `ovalues[node_id] = ivalues[node_id] + (dt/tau) * (-ivalues[node_id] + z)`, reading the previous step's value from the correct buffer.
   - Test update: `test/test_ctrnn.jl` analytical comparison test was calibrated against the buggy `2*tau` effective time constant. Updated to use correct `tau`. All single-step hand-computed tests were unaffected (both buffers start at zero).

2. **IZNN spike threshold `>` vs `>=`** (`src/iznn.jl:110`)
   - Root cause: Code used `v > 30.0` but the Izhikevich (2003) paper, the docstring (line 9), and neat-python all use `v >= 30`.
   - Fix: Changed to `v >= 30.0`.

3. **BoolAttribute mutation randomizes instead of flipping** (`src/attributes.jl:145`)
   - Root cause: `rand(rng, Bool)` gives 50/50 chance of true/false, so a mutation has 50% chance of producing no change. neat-python flips the value. This halved the effective mutation rate.
   - Fix: Changed to `!value` (deterministic flip).
   - Test update: `test/test_attributes.jl` basic mutation test updated to verify deterministic flip in both directions.

4. **Missing `post_evaluate!` default for Reporter** (`src/reporting.jl:23`)
   - Root cause: `start_generation!`, `end_generation!`, `found_solution!`, and `complete_extinction!` all had default no-op methods. `post_evaluate!` had only a comment saying "must be implemented by each reporter that needs it", but `population.jl` calls it unconditionally on all reporters. Any custom Reporter subtype would crash with MethodError.
   - Fix: Added default no-op method with matching signature.

5. **Wrong typo correction mapping** (`src/validation.jl:165`)
   - Root cause: `TYPO_CORRECTIONS` mapped `:max_generations => :fitness_threshold`. These are semantically unrelated. A user writing `max_generations` wants generation count control, not a fitness threshold.
   - Fix: Changed to `:max_generations => :no_fitness_termination` which is the closest semantic match (controls whether evolution runs for a fixed number of generations).

6. **Stagnation uses `maximum` regardless of fitness function** (`src/stagnation.jl:35`)
   - Root cause: `prev_fitness` was always `maximum(s.fitness_history)` regardless of the configured `species_fitness_func`. If the fitness function was `mean`, a species would only be marked as improved if its mean exceeded its historical *maximum* mean -- a strictly harder criterion than "improved over last generation."
   - Fix: Changed to `last(s.fitness_history)` so improvement is measured against the most recent generation's fitness value computed by the same function.
   - Alternative considered: Could have used `stagnation.species_fitness_func(s.fitness_history)` but that changes the semantics too much. `last()` is the natural choice: "did the species improve compared to last generation?"

7. **`connect_fs_neat!` implements full connectivity** (`src/genome.jl:105-129`)
   - Root cause: FS-NEAT (Feature-Selective NEAT) should connect a single random input to outputs, letting evolution discover which inputs matter. The implementation connected *all* inputs to all outputs, making it identical to `connect_full!`.
   - Fix: Select one random input via `rand(rng, 1:length(config.input_keys))` and connect only that input.
   - Test update: `test/runtests.jl` FS-NEAT connection count assertions updated (2->1, 2->1, 6->4).

8. **Crossover with `nothing` fitness prefers wrong parent** (`src/genome.jl:196-203`)
   - Root cause: The original condition `parent1.fitness === nothing || parent2.fitness === nothing` fell through to treat parent1 as fitter. If parent1 had `nothing` fitness but parent2 had a real fitness, parent1 was incorrectly treated as fitter.
   - Fix: Expanded to four explicit cases: both `nothing`, only parent1 `nothing`, only parent2 `nothing`, equal fitness. The parent with real fitness is now correctly treated as fitter.
